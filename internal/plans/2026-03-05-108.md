# Bug Investigation Plan - mise in GitHub Actions

## Summary

The reusable workflows were partially migrated from framework-specific setup steps (`ruby/setup-ruby`, `actions/setup-node`, direct Rails/Yarn/PNPM commands) to `mise`, but the migration is incomplete and inconsistent with the official mise GitHub Actions pattern.

Current workflows pin `jdx/mise-action@v2` and then run a separate `mise install` step. This is a fragile path and is the reported failure point. The official generated workflow pattern from mise (`mise generate github-action`) uses:

- `jdx/mise-action@v3`
- `MISE_EXPERIMENTAL: true`
- `mise run <task>` as the execution primitive

The repo also still encodes Rails- and package-manager-specific commands directly in workflows, which prevents universal reuse across Rails/Nuxt/Next.js repos and conflicts with the requirement to standardize on `mise run <task>`.

## Root cause analysis

### What the code is supposed to do

Reusable workflows should:

1. Prepare toolchains in CI with mise reliably.
2. Execute project automation through standardized mise tasks (`mise run setup`, `mise run test`, `mise run lint`, etc.).
3. Stay framework-agnostic so downstream repos define behavior in their own `mise.toml`.

### What it is actually doing

Across all workflows (`ci.yml`, `deploy.yml`, `prettier.yml`, `standard.yml`, `erb-lint.yml`, `brakeman.yml`):

- Uses `jdx/mise-action@v2`
- Then runs a manual `mise install`
- Then runs framework/tool specific commands directly (for example: `bundle install`, `bin/rails ...`, `pnpm install`, `pnpm deploy`, reviewdog actions)

This means workflow behavior is not task-driven and not universal.

### Exact root causes

1. **Non-canonical mise CI usage in workflows**  
   All workflows use an older action major (`@v2`) and a separate install invocation pattern rather than the current generated pattern (`@v3` + task execution). The failure is reported at `mise install`, and that step appears in every workflow as a standalone command, making this a single systemic failure mode.

2. **Migration stopped at tool installation, not task execution**  
   The command layer remained framework-specific (Rails/PNPM/reviewdog action wrappers), so workflows are tightly coupled to app internals instead of delegating to repo-owned mise tasks.

3. **Documentation gap causes inconsistent downstream setup**  
   `README.md` does not define required `mise.toml` task contracts for Rails/Nuxt/Next.js. Without explicit task/interface expectations, downstream repos cannot consistently satisfy reusable workflow assumptions.

### Regression assessment

This appears to be a **regression introduced in commit `9e2a6b3` ("Use mise for dependencies")**, where setup actions were replaced but execution contracts were not redesigned around mise tasks.

### Related bugs sharing same root cause

- Any current/future workflow that shells out to framework-specific commands instead of `mise run <task>` will drift and fail inconsistently across repo types.
- Any workflow that keeps a separate `mise install` step (instead of relying on action-managed install and then running tasks) inherits the same installation failure surface.

## Files to modify (with specific suggested changes)

### `.github/workflows/ci.yml`

- Replace the current install pattern with modern mise action usage:
  - Switch `jdx/mise-action@v2` to `jdx/mise-action@v3`
  - Remove explicit `run: mise install` step
  - Add `env: MISE_EXPERIMENTAL: true` at workflow/job scope for parity with generated workflow pattern
- Replace framework-specific steps with task calls:
  - `Install system dependencies` -> move into `mise run setup` task (or split into `setup` and `setup:system` if needed)
  - `bundle install`, DB setup, optional Node install, asset precompile, tests -> replace with `mise run ci` (or `mise run setup` then `mise run test`)
- Keep service containers (postgres/redis) in workflow, but hand orchestration to tasks.
- Keep artifact upload, but trigger on failure after task run.

### `.github/workflows/deploy.yml`

- Update mise action usage as above (`@v3`, remove standalone `mise install`, set `MISE_EXPERIMENTAL`).
- Replace direct dependency and deploy commands with:
  - `mise run setup`
  - `mise run deploy`
- Keep `SHOPIFY_CLI_PARTNERS_TOKEN` and `COMMIT_URL` env passthrough; task should consume these.

### `.github/workflows/prettier.yml`

- Update mise action usage as above.
- Replace direct pnpm/cache/install + prettier action invocation with task-driven command:
  - `mise run lint` or `mise run format:check` (decide final naming contract)
- If auto-format-on-PR behavior is still desired, define a dedicated task (for example `format:write`) and run it explicitly.
- Re-evaluate whether `creyD/prettier_action` is still needed once task-driven flow is in place.

### `.github/workflows/standard.yml`

- Update mise action usage as above.
- Replace direct bundler setup + `kirillplatonov/action-standard` with `mise run lint:ruby` (or `mise run lint` if consolidated).
- If reviewdog comments are required, ensure task output format remains compatible or use a thin wrapper step that runs `mise run lint:ruby`.

### `.github/workflows/erb-lint.yml`

- Update mise action usage as above.
- Replace direct bundler + `tk0miya/action-erblint` with `mise run lint:erb` (or unified lint task).
- Keep PR reporting strategy if needed via wrapper action, but primary command should be task-based.

### `.github/workflows/brakeman.yml`

- Update mise action usage as above.
- Replace direct bundler + reviewdog brakeman action with `mise run security` (or `mise run lint:security`).
- If reviewdog integration is required, keep minimal wrapper that executes task output, not direct tool command.

### `README.md`

- Add a new section describing the CI task contract expected by reusable workflows:
  - Required tasks: `setup`, `ci`, `test`, `lint`, `build`, `dev`
  - Optional tasks: `deploy`, `security`, `format`, `format:write`, framework-specific subtasks
- Add framework-specific `mise.toml` examples for:
  - Rails
  - Nuxt
  - Next.js
- Include placeholders where command choice is uncertain (requested):
  - Rails lint placeholder (for example: `# TODO: choose rubocop/standardrb command`)
  - Framework-specific build/deploy placeholders as needed
- Update manual setup examples to call reusable workflows that execute `mise run <task>` instead of framework commands.

## Testing approach

1. **Static validation**
   - Run workflow linting locally (if available) and ensure YAML syntax and reusable-workflow interfaces remain valid.

2. **Task contract validation in sample repos**
   - Rails sample repo: verify `mise run setup`, `mise run ci`, `mise run test`, `mise run lint`
   - Nuxt sample repo: same contract
   - Next.js sample repo: same contract

3. **GitHub Actions validation**
   - Trigger each reusable workflow from a caller repo using `workflow_dispatch` and PR events:
     - `ci.yml` with and without node-related tasks
     - `prettier.yml` / lint workflow
     - `deploy.yml` with mock or dry-run deploy task where possible

4. **Failure-path validation**
   - Confirm failed tasks return non-zero and surface clear logs.
   - Confirm artifact upload and reporting behavior still works after task-based conversion.

## Risks

1. **Task naming mismatch across repos**  
   If downstream repos do not implement required task names, workflows will fail fast. Mitigate by documenting a strict minimal task contract in README.

2. **Loss of action-specific review annotations**  
   Replacing reviewdog wrapper actions with pure task runs may reduce inline PR annotations. Mitigate by keeping thin wrappers where annotation UX is required.

3. **Framework edge cases hidden inside old workflows**  
   Existing Rails-specific sequencing (DB setup, assets, env vars) may be implicitly relied on. Mitigate by encoding that logic in framework-specific `mise.toml` tasks and validating in at least one repo per framework.

4. **Install/cache behavior changes during action upgrade**  
   Moving from `jdx/mise-action@v2` to `@v3` changes defaults (for example cache key prefix version). Mitigate by validating cache hits/misses across two consecutive runs.
